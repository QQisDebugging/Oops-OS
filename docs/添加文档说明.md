# 文档编写规范

本项目的设计文档必须做到“可读、可复现、可验证”。参考获奖项目的写法，每个功能模块的文档建议统一采用 **背景 + 设计 + 关键代码 + 测试 + 结果/结论** 的结构，并且写出“为什么这样做、如何做、在哪里做、怎么验证”。

---

## 1. 推荐结构（每个功能/系统调用）

### 1.1 背景（为什么做）
- 问题或痛点是什么？（例如：死锁、丢唤醒、羊群效应）
- 期望达到的目标/语义是什么？
- 与 xv6 或已有实现的差异与不足是什么？

### 1.2 设计（怎么做）
- 数据结构：新增字段、关键状态、全局表（说明字段含义）
- 算法流程：核心步骤、关键路径与异常处理
- 同步策略：用到的锁/睡眠通道、避免竞态的手段
- 边界条件：失败路径、资源不足、异常退出等

### 1.3 关键代码（在哪里做）
- 只贴关键片段，不贴整段实现。
- 片段必须能定位：注明文件路径、函数名。
- 片段前后说明：这段代码负责什么、为何关键。

### 1.4 测试（怎么验证）
- 测试点列表：正常路径/异常路径/竞争路径。
- 对比逻辑：有无改进时的差异（例如死锁 vs AND 成功）。
- 运行方式：命令 + 预期输出。

### 1.5 结果/结论（效果如何）
- 是否满足语义？是否避免了原问题？
- 性能或稳定性是否有改善？
- 尚未覆盖的风险/后续可改进点。

---

## 2. 写作细节要求

1. **保持一致性**  
   全部模块都用同一结构与标题层级，便于评分与查阅。

2. **代码片段最小化**  
   只贴能解释逻辑的关键片段；大片重复代码会降低可读性。

3. **路径明确**  
   每段关键代码要标注路径，如：  
   `kernel/sysproc.c::sys_sem_p()`。

4. **测试可复现**  
   必须写出具体命令，例如：  
   `usertests deadlocktest` 或 `deadlocktest`。

5. **结果可对比**  
   需要写出典型输出或指标变化，体现“改进有效”。

## 4. 与评分点对齐

- **代码实现（30%）**：文档需说明实现思路与关键路径。
- **文档质量（30%）**：结构完整、逻辑清晰、可复现。
- **教学价值（10%）**：突出原理与机制。
- **工程规范（10%）**：接口说明清楚、注释与文档一致。
- **答辩展示（20%）**：有可演示的测试与对比结果。
