# OopsOS——设备管理

OopsOS 目前的设备管理主要覆盖控制台/串口、块设备、网络设备，并新增了一个可复用的 spooling 机制，用于演示“生产者与设备解耦”的输出流程。

## 1. 总览
- **字符设备：** 控制台（console/uart）与 spooling。
- **块设备：** virtio 磁盘，配合 buffer cache 完成块 I/O。
- **网络设备：** QEMU 模拟的 E1000 网卡，支持基础 UDP/IP 通信。

## 2. 控制台/串口（Console + UART）
控制台作为用户态与内核交互的主要入口，基于 UART 实现字符输入/输出。内核提供 console 输入缓冲区，串口中断触发时将字符写入缓冲区；读操作从缓冲区取出字符，写操作通过 UART 发送。

关键点：
- **中断驱动：** UART 接收中断唤醒读者，避免忙等。
- **缓冲与同步：** console 使用环形缓冲区，读写通过锁与 sleep/wakeup 进行同步。
- **与 shell 交互：** `init` 启动 `sh` 以后，控制台承担交互式输入输出。

## 3. 块设备与缓冲（Virtio + Buffer Cache）
OopsOS 通过 virtio 块设备完成磁盘 I/O，`bio`/`fs` 负责块缓存与文件系统读写：
- **buffer cache：** 统一缓存磁盘块，避免频繁 I/O。
- **日志与一致性：** 与文件系统日志模块协作，保证元数据一致性。

## 4. 网络设备（E1000）
OopsOS 是在 xv6 基础上改进的内核。在 xv6 原有网络基础上，完善了 E1000 网卡驱动，通过该驱动可发送和接收数据包，提供对 `UDP/IP` 协议的简单支持。
E1000 由 QEMU 提供模拟设备，已在构建配置中启用用户态网络栈与 E1000 网卡。
E1000 驱动中的 `transmit()` 和 `recv()` 由我们实现，其余框架沿用 xv6 代码。

### 4.1 E1000 的交互与数据结构
E1000 使用 DMA 技术，将接收的数据直接写入内存，这在大数据量场景下可以显著降低 CPU 负担。
网络数据包使用 `mbuf` 描述，支持链表组合以处理较大的报文。

```c
struct mbuf {
  struct mbuf *next;   // 链表指针
  char *head;          // 数据起始地址
  unsigned int len;    // 数据长度（字节）
  char buf[MBUF_SIZE]; // 实际数据缓冲区
};
```

### 4.2 接收流程
网卡收到数据后触发中断，驱动在中断处理路径中轮询 RX 环形队列，将完整包交给上层网络栈。

#### 4.2.1 接收描述符
```c
struct rx_desc {
  uint64 addr;    // 描述符数据缓冲区地址
  uint16 length;  // DMA 写入长度
  uint16 csum;    // 校验和
  uint8 status;   // 描述符状态
  uint8 errors;   // 错误标记
  uint16 special; // 特殊字段
};
```

#### 4.2.2 接收环形队列
接收采用环形队列（ring buffer）管理描述符：
- `head` 指向设备将写入的位置；
- `tail` 指向软件处理位置（通常是最旧未处理包的下一个描述符）。

<img src="../img/10.png" alt="10" style="zoom:50%;" />

设备将数据写入 `head` 所在的描述符缓冲区，并推进 `head`；
软件按 `tail+1` 位置顺序读取、交付并推进 `tail`。

#### 4.2.3 `e1000_recv()` 实现要点
接收时循环扫描 RX 队列，直到遇到未完成描述符：
```c
static void
e1000_recv(void)
{
  while (1) {
    uint32 ind = (regs[E1000_RDT] + 1) % RX_RING_SIZE;
    struct rx_desc *desc = &rx_ring[ind];
    if (!(desc->status & E1000_RXD_STAT_DD))
      return;

    rx_mbufs[ind]->len = desc->length;
    net_rx(rx_mbufs[ind]);

    rx_mbufs[ind] = mbufalloc(0);
    desc->addr = (uint64)rx_mbufs[ind]->head;
    desc->status = 0;
    regs[E1000_RDT] = ind;
  }
}
```

### 4.3 发送流程
发送时驱动将 `mbuf` 填入 TX 描述符并通知设备发送，完成后回收对应缓冲。

#### 4.3.1 发送描述符
```c
struct tx_desc {
  uint64 addr;   // 发送缓冲区地址
  uint16 length; // 数据长度
  uint8 cso;     // 校验和偏移
  uint8 cmd;     // 发送命令
  uint8 status;  // 描述符状态
  uint8 css;     // 校验和控制
  uint16 special;
};
```

#### 4.3.2 发送环形队列
发送队列同样使用环形结构：
- `head` 指向等待时间最长的待发数据；
- `tail` 指向新数据插入位置。

<img src="../img/11.png" alt="11" style="zoom:50%;" />

#### 4.3.3 `e1000_transmit()` 实现要点
```c
int e1000_transmit(struct mbuf *m)
{
  acquire(&e1000_lock);
  uint32 ind = regs[E1000_TDT];
  struct tx_desc *desc = &tx_ring[ind];
  if (!(desc->status & E1000_TXD_STAT_DD)) {
    release(&e1000_lock);
    return -1;
  }
  if (tx_mbufs[ind]) {
    mbuffree(tx_mbufs[ind]);
    tx_mbufs[ind] = 0;
  }
  desc->addr = (uint64)m->head;
  desc->length = m->len;
  desc->cmd = E1000_TXD_CMD_EOP | E1000_TXD_CMD_RS;
  tx_mbufs[ind] = m;
  regs[E1000_TDT] = (regs[E1000_TDT] + 1) % TX_RING_SIZE;
  release(&e1000_lock);
  return 0;
}
```

## 5. Spooling（新增）
Spooling 机制用于模拟“打印/输出设备缓冲”的场景：生产者只需把数据写入 spooling 设备，由专门的 spooler 进程负责从设备读取并写入目标文件，从而实现**生产者与设备输出解耦**。

### 5.1 实现策略
- **字符设备驱动：** `kernel/driver/spool.c` 实现一个内核环形缓冲区。
- **设备注册：** 在 `devsw` 中注册 `SPOOL` 的 read/write，并在 `main()` 调用 `spoolinit()` 完成初始化。
- **同步机制：** 读写使用 `sleep/wakeup`，缓冲区满时写阻塞，空时读阻塞。

### 5.2 使用方式
spool 作为设备文件被创建并访问：
```c
mknod("spool", SPOOL, 0);
int fd = open("spool", O_RDWR);
write(fd, "job...", ...);
```
专用的 spooler 进程从 `spool` 读出数据并写入 `spool.out`，不需要新增系统调用。

### 5.3 测试程序
`user/test/spooltest.c` 验证 spooling 工作流：
- 父进程创建并打开 `spool` 设备，写入两段消息与结束标记。
- 子进程作为 spooler 读取 `spool` 并写入 `spool.out`。
- 父进程校验输出内容一致性，打印 `spooltest: ok`。
