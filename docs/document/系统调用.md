# **define SYS_cps 22**

- **功能：遍历所有进程控制块，并打印相关信息**
- **输出：成功执行，返回22（系统调用号）**

**背景：**  
调度/并发相关实验中，需要快速查看当前所有进程的状态与优先级信息。`cps` 作为调试工具系统调用，提供类似 `ps` 的最小化快照，便于观察调度策略差异（DYNPRIO/MLFQ）。

**设计：**

1. 遍历 `proc[NPROC]` 进程表，输出 PID、状态、优先级信息。
2. 根据调度策略输出不同字段：
   - `SCHED_DYNPRIO`：输出 `dyn_priority`
   - `SCHED_MLFQ`：输出 `mlfq_level/mlfq_ticks`
3. 该接口仅用于调试，不对输出做强一致性保证（避免持锁导致死锁）。

**关键代码：**

```c
int cps(void)
{
  struct proc *p = proc;
#if defined(SCHED_DYNPRIO)
  printf("name \t pid \t state \t \t priority \tdyn_priority\n");
#else
  printf("name \t pid \t state \t \t priority \tmlfq_lvl \tmlfq_ticks\n");
#endif
  for (p = proc; p < &proc[NPROC]; p++)
  {
    if (p->state == SLEEPING)
      printf("%s \t %d \t SLEEPING \t %d\t \t%d\t \t%d\n",
             p->name, p->pid, p->priority, p->mlfq_level, p->mlfq_ticks);
    else if (p->state == RUNNING)
      printf("%s \t %d \t RUNNING \t %d\t \t%d\t \t%d\n",
             p->name, p->pid, p->priority, p->mlfq_level, p->mlfq_ticks);
    else if (p->state == RUNNABLE)
      printf("%s \t %d \t RUNNABLE \t %d\t \t%d\t \t%d\n",
             p->name, p->pid, p->priority, p->mlfq_level, p->mlfq_ticks);
  }
  return 22;
}
```

**测试：**

- 运行 `currentproc`（调用 `cps()`）观察输出字段是否与编译调度策略匹配。
- 在 `SCHED=MLFQ` 和 `SCHED=DYNPRIO` 两种构建下对比字段变化。

# **define SYS_trace 23**

- **功能：为当前进程设置系统调用跟踪掩码**
- **输入：bitmask（每一位对应一个系统调用号）**
- **输出：成功返回 0，失败返回 -1**

**背景：**  
系统调用跟踪是调试与教学常用工具。通过 `trace` 可以观察进程执行了哪些系统调用、返回值如何，方便排查异常路径或性能问题。

**设计：**

1. 在 `struct proc` 中维护 `trace_mask` 位图。
2. 用户设置后，内核在 `syscall()` 中检查该位是否置位：
   - 若置位则打印 `pid + syscall name + return value`。
3. 只影响调用 `trace()` 的进程，不影响全局。

**关键代码：**

```c
uint64 sys_trace(void)
{
  int n;
  if (argint(0, &n) < 0)
    return -1;
  myproc()->trace_mask = n;
  return 0;
}

if ((p->trace_mask & (1 << num)) != 0) {
  printf("%d: syscall %s -> %d", p->pid, syscall_name, p->trapframe->a0);
}
```

**测试：**

- 运行 `trace` 用户程序，例如：`trace 1 ls`，观察 `SYS_fork/SYS_exec/SYS_read` 等调用输出。
- 结合 `usertests` 中的单个用例，验证 trace 输出是否对应预期调用路径。

# **define SYS_sysinfo 24**

- **功能：查看当前进程数量、剩余空闲内存，并返回 swap buffer 命中统计**
- **输入：系统资源信息**
- **输出：执行成功返回0**

**实现思路：**

1. **定义存储系统信息结构体并填充信息**
  - **定义了sysinfo info；**
  - **调用freebytes函数将获取到的内存信息填充**
  - **调用procnum函数，填充进程数量**
2. **获取目标用户的空间地址**
  - **调用argaddr(0, &dstaddr);将传入的用户空间地址的第0个地址传入dstaddr**
3. **将系统信息复制到用户空间**
    **-使用 copyout 函数尝试将内核空间中 info 结构体所存储的系统信息（包含了前面获取到的空闲内存、进程数量等内容）复制到用户空间中由 dstaddr 指定的地址处。**

**代码实现：**

```c
uint64 sys_sysinfo(void)
{
  struct sysinfo info;
  freebytes(&info.freemem);
  procnum(&info.nproc);
  swap_pbuf_stats(&info.swapbuf_hits, &info.swapbuf_misses,
                  &info.swapbuf_cached);

  uint64 dstaddr;
  argaddr(0, &dstaddr);
  if (copyout(myproc()->pagetable, dstaddr, (char *)&info, sizeof info) < 0)
    return -1;
  return 0;
}
```

**测试：**

- 运行 `sysinfo` 用户程序，观察输出的 `freemem/nproc` 是否合理变化。
- 在大量 `fork` 后再执行 `sysinfo`，验证进程数变化。

# **define SYS_sbrk 12**

- **功能：使用懒分配，完成对进程内存空间大小的调整**
- **输入：进程内存的调整量**
- **输出：执行成功返回起始地址，失败返回-1**

**实现思路：**

1. **获取参数及错误处理**
  - **调用argint获取用户空间传入的第0个参数，并赋值给n**
2. **获取当前进程相关信息**
  - **调用myproc函数获取当前正在运行的进程的结构体指针p进而取得进程的相关信息**
3. **对内存空间的增减操作**
  - **根据传入的n的正负值对进程内存进行懒分配，执行相应增减操作**
4. **非法处理**
  - **如果在减少内存空间时，计算得到的新内存大小不大于 0（即 sz + n <= 0），这种情况被视为非法的内存调整操作，函数直接返回 -1，表示系统调用执行失败**
5. **返回调整前的内存起始地址**
  - **最后，无论内存是增加还是减少操作，只要前面的操作没有出现错误（返回 -1 的情况），函数都会返回 addr，也就是记录的调整前的内存起始地址，这个返回值可以让用户空间程序知晓之前内存的起始位置情况**

**代码实现：**

```c
uint64
sys_sbrk(void)
{
  int addr;
  int n;

  if (argint(0, &n) < 0)
    return -1;

  struct proc *p = myproc();
  addr = p->sz;
  uint64 sz = p->sz;

  if (n > 0)
  {
    // 懒分配
    p->sz += n;
  }
  else if (sz + n > 0)
  {
    sz = uvmdealloc(p->pagetable, sz, sz + n);
    p->sz = sz;
  }
  else
  {
    return -1;
  }
  return addr;
}
```

# **define SYS_setPriority 25**

- **功能：接收用户传入的进程标识符和优先级数值参数，对进程优先级进行设置**
- **输入：用户空间传入的整形参数**
- **输出：执行成功返回0，失败返回-1**

**背景：**  
在优先级调度（DYNPRIO/MLFQ）实验中，需要人工调整某个进程的“静态优先级”，以观察调度响应与公平性变化。`setPriority` 提供一个最小控制接口。

**设计：**

1. 通过 `pid` 定位目标进程，更新其 `priority` 字段。
2. 对不同调度策略做一致性处理：
   - `SCHED=DYNPRIO`：`priority` 作为基准，动态优先级在调度时计算。
   - `SCHED=MLFQ`：`priority` 映射为初始队列层级，并重置 `mlfq_ticks`。
3. 允许优先级范围 [0, 20]，超出则截断。

**关键代码：**

```c
uint64 sys_setPriority(void)
{
  int pid, priority;

  // 从用户栈中读取 pid 和 priority 参数
  if (argint(0, &pid) < 0 || argint(1, &priority) < 0)
  {
    return -1; // 参数读取失败，返回错误
  }

  // 调用 setPriority 函数设置进程优先级
  return setPriority(pid, priority);
}
```

**测试：**

- 运行 `setp <pid> <priority>` 调整进程优先级，观察 `cps` 输出变化。
- 在 `SCHED=MLFQ` 与 `SCHED=DYNPRIO` 下分别测试，验证映射策略是否生效。

# **define SYS_execve 26**

- **功能：执行指定路径的程序**
- **输入：获取外部输入的地址与命令**
- **输出：执行成功调用exec函数执行指定程序，执行失败返回-1**
- **补充说明：** 当前内核采用按需装入，`execve` 不再一次性加载 ELF 段，运行时首次访问对应页触发缺页再加载；仅用户栈在 `exec` 阶段预先分配。

**实现思路：**

1. **参数获取与初始化**
  - **定义用于存储程序路径的字符数组 path 和指向参数字符串的指针数组 argv**
  - **通过 argstr 和 argaddr 函数按顺序获取程序路径和用户空间参数列表的起始地址，只要其中一个获取操作失败，就直接返回 -1，不再继续后续流程**
  - **取成功后，将程序路径 path 赋值给 argv[0]，作为参数列表的第一个元素，也就是将要执行程序的路径本身**
2. **构建参数列表**
  - **进入循环来处理参数列表中的其他参数（从索引 1 开始），循环条件为持续进行直到遇到结束条件break退出**
  - **在循环中，先进行边界检查，如果参数索引 i 超出了 argv 数组所能容纳的最大元素个数（通过 NELEM(argv) 判断），则跳转到错误处理部分（bad）**
  - **接着使用 fetchaddr 函数获取每个参数在用户空间的实际地址（通过计算偏移量 uargv + sizeof(uint64) * (i - 1) 来定位每个参数的地址**
  - **当获取到的参数地址对应的 uarg 值为 0 时，意味着参数列表结束，将当前 argv 元素赋值为 0 来标记参数列表的结尾，然后跳出循环**
  - **对于每个非空的参数，先调用 kalloc 函数为其分配内核内存空间（用于存放从用户空间获取的参数字符串内容），若内存分配失败则跳转到错误处理部分。分配成功后，使用 fetchstr 函数从用户空间对应的地址 uarg 处获取字符串内容，并存储到刚分配好的内存中（通过 argv[i] 指向的内存区域）**
  3. **程序执行与内存清理**

- **完成参数列表构建后，调用 exec(path, argv) 函数来执行指定的程序，传递构建好的程序路径和参数列表进去**
- **在 exec 函数执行完毕后（无论执行成功与否），需要清理之前为参数分配的内存（除了 argv[0] 对应的程序路径，因为它不需要清理，从索引 1 开始清理），通过循环逐个调用 kfree 函数释放内存。最后返回 exec 函数的返回值作为系统调用的最终结果返回给调用者**

  4. **错误处理**

- **代码中有一个标记为 bad 的错误处理部分，当在参数处理过程中出现各种错误（如内存分配失败、获取字符串内容失败等）时会跳转到此处。在这里同样通过循环逐个调用 kfree 函数清理已经为参数分配的内存（只要参数不为空），然后返回 -1，表示系统调用执行失败，以此来保证在出现异常情况时能合理地释放资源并向调用者反馈错误信息**

**代码实现：**

```c
uint64
sys_execve(void)
{
  char path[260], *argv[MAXARG];
  int i;
  uint64 uargv, uarg;
  // 获取路径和参数地址
  if (argstr(0, path, 260) < 0 || argaddr(1, &uargv) < 0)
  {
    return -1;
  }
  memset(argv, 0, sizeof(argv));
  // 设置 argv[0] 为程序的路径
  argv[0] = path;
  // 从用户空间获取其他参数
  for (i = 1;; i++)
  {
    if (i >= NELEM(argv))
      goto bad;
    // 获取每个参数的地址
    if (fetchaddr(uargv + sizeof(uint64) * (i - 1), (uint64 *)&uarg) < 0)
      goto bad;
    // 如果参数为空，结束循环
    if (uarg == 0)
    {
      argv[i] = 0;
      break;
    }
    // 为参数分配内存
    argv[i] = kalloc();
    if (argv[i] == 0)
      goto bad;
    // 获取字符串内容
    if (fetchstr(uarg, argv[i], PGSIZE) < 0)
      goto bad;
  }
  // 打印参数（调试用）
  // for (i = 0; i < NELEM(argv) && argv[i] != 0; i++)
  // {
  //   printf("argv[%d]: %s\n", i, argv[i]);
  // }
  // 调用 exec 执行程序
  // printf("%s", path);
  int ret = exec(path, argv);
  // 清理已分配的内存
  for (i = 1; i < NELEM(argv) && argv[i] != 0; i++)
    kfree(argv[i]);
  return ret;
bad:
  // 清理已分配的内存
  for (i = 0; i < NELEM(argv) && argv[i] != 0; i++)
    kfree(argv[i]);
  return -1;
}
```

# **define SYS_getparentpid 27**

- **功能：获取当前进程父进程的pid**
- **输入：指向当前进程地址的指针**
- **输出：父进程pid**

**背景：**  
进程树与父子关系是操作系统核心概念。获取父进程 PID 便于实现“父子协作”、调试进程关系，或进行简单的进程家族管理。

**设计：**

1. 通过 `myproc()` 获取当前进程。
2. 直接读取 `p->parent->pid` 返回。
3. 若父进程为 `init`，返回 init 的 PID。

**关键代码：**

```c
uint64 sys_getparentpid(void)
{
  struct proc *p = myproc();
  return p->parent->pid;
}
```

**测试：**

- 运行 `getparentpid` 用户程序，在父进程/子进程中分别打印返回值。
- 在 `fork` 后子进程调用该接口，验证父 PID 与 `getpid()` 匹配。

# **define SYS_mmap 29**

- **功能：建立文件映射**
- **输入：内存映射的具体要求与配置情况**
- **输出：执行成功返回映射区域在进程虚拟地址空间中的起始虚拟地址，执行失败返回预先定义错误err**
- **补充说明：** 映射页采用缺页按需加载；`MAP_SHARED` 的文件映射页在置换时会写回文件并回收物理页，后续访问由缺页重新从文件载入。

**实现思路：**

1. **参数定义与获取**
  - **首先定义了一系列变量用于存储系统调用传入的参数，包括 addr（期望的映射起始虚拟地址）、length（映射区域的长度）等相关信息**
  - **通过多个函数（argaddr、argint、argfd 等）从外部获取系统调用的参数，并进行错误判断**
2. **参数合法性检查**
  - **对 addr、offset 和 length 参数做进一步的合法性判断**
3. **权限与文件可写性检查**
  - **检查传入的参数中关于内存保护权限（prot）、映射标志（flags）以及文件可写性（vfile->writable）之间的兼容性**
4. **虚拟内存空间检查**
  - **获取当前进程的结构体指针 p = myproc()，然后检查当前进程的虚拟内存空间是否足够容纳要映射的区域**
5. **虚拟内存区域分配与设置**
  - **通过循环遍历查找当前进程中未使用的虚拟内存区域结构体**
  - **当找到一个未使用的 VMA 结构体（p->vma[i].used == 0）时,标记改VMA结构体已使用，并对映射区域进行配置，再通过filedup增加关联文件引用技术，最后更新当前进程已使用内存大小，返回该VMA结构体的其实虚拟机地址**
6. **映射失败情况处理**
  - **如果遍历完所有的 VMA 结构体都没有找到未使用的，说明无法为此次映射分配合适的虚拟内存区域，那么函数就返回表示错误的 err 值**

**代码实现：**

```c
uint64
sys_mmap(void)
{
  uint64 addr;
  int length;
  int prot;
  int flags;
  int vfd;
  struct file *vfile;
  int offset;
  uint64 err = 0xffffffffffffffff;

  // 获取系统调用参数
  if (argaddr(0, &addr) < 0 || argint(1, &length) < 0 || argint(2, &prot) < 0 ||
      argint(3, &flags) < 0 || argfd(4, &vfd, &vfile) < 0 || argint(5, &offset) < 0)
    return err;

  // 实验提示中假定addr和offset为0，简化程序可能发生的情况
  if (addr != 0 || offset != 0 || length < 0)
    return err;

  // 文件不可写则不允许拥有PROT_WRITE权限时映射为MAP_SHARED
  if (vfile->writable == 0 && (prot & PROT_WRITE) != 0 && flags == MAP_SHARED)
    return err;

  struct proc *p = myproc();
  // 没有足够的虚拟地址空间
  if (p->sz + length > MAXVA)
    return err;

  // 遍历查找未使用的VMA结构体
  for (int i = 0; i < NVMA; ++i)
  {
    if (p->vma[i].used == 0)
    {
      p->vma[i].used = 1;
      p->vma[i].addr = p->sz;
      p->vma[i].len = length;
      p->vma[i].flags = flags;
      p->vma[i].prot = prot;
      p->vma[i].vfile = vfile;
      p->vma[i].vfd = vfd;
      p->vma[i].offset = offset;

      // 增加文件的引用计数
      filedup(vfile);

      p->sz += length;
      return p->vma[i].addr;
    }
  }

  return err;
}
```

# **define SYS_munmap 30**

- **功能：用于取消文件映射**
- **输入：需要取消映射的其实虚拟地址以及需要取消映射的长度**
- **输出：执行成功返回0，异常返回-1**

**实现思路：**

1. **参数获取**
  - **定义两个变量addr、length分别表示要取消映射的起始虚拟地址和要取消映射区域长度。并通过argaddr、argint两个函数从外部获取，并对获取失败进行-1返回**
2. **遍历虚拟内存区域（VMA）查找匹配项**
  - **获取当前进程的结构体指针 p = myproc()**
  - **循环遍历当前进程中所有的虚拟内存区域结构体（VMA）**
  - **对每个VMA进行条件判断，p->vma[i].used为真，p->vma[i].len >= length**
3. **根据地址范围调整 VMA 信息**
  - **从起始地址开始**
  - **从结束位置开始**
4. **处理未找到匹配 VMA 的情况**
  - **若遍历完成i == NVMA表示没有匹配项，返回-1**
5. **将共享可写页面写回文件系统**
  - **如果找到的 VMA 满足是共享映射（p->vma[i].flags == MAP_SHARED）并且具有可写权限（(p->vma[i].prot & PROT_WRITE)!= 0），则调用 filewrite 函数将对应页面的内容写回关联的文件系统中**
6. **解除页面映射**
  - **调用 uvmunmap 函数来解除从起始地址 addr 开始，长度为 length / PGSIZE 个页面**
7. **清理 VMA 相关资源**
8. **返回操作结果**

**代码实现：**

```c
uint64
sys_munmap(void)
{
  uint64 addr;
  int length;
  if (argaddr(0, &addr) < 0 || argint(1, &length) < 0)
    return -1;

  int i;
  struct proc *p = myproc();
  for (i = 0; i < NVMA; ++i)
  {
    if (p->vma[i].used && p->vma[i].len >= length)
    {
      // 根据提示，munmap的地址范围只能是
      // 1. 起始位置
      if (p->vma[i].addr == addr)
      {
        p->vma[i].addr += length;
        p->vma[i].len -= length;
        break;
      }
      // 2. 结束位置
      if (addr + length == p->vma[i].addr + p->vma[i].len)
      {
        p->vma[i].len -= length;
        break;
      }
    }
  }
  if (i == NVMA)
    return -1;

  // 将MAP_SHARED页面写回文件系统
  if (p->vma[i].flags == MAP_SHARED && (p->vma[i].prot & PROT_WRITE) != 0)
  {
    filewrite(p->vma[i].vfile, addr, length);
  }

  // 判断此页面是否存在映射
  uvmunmap(p->pagetable, addr, length / PGSIZE, 1);

  // 当前VMA中全部映射都被取消
  if (p->vma[i].len == 0)
  {
    fileclose(p->vma[i].vfile);
    p->vma[i].used = 0;
  }

  return 0;
}
```

# **define SYS_sem_p 35**

- **功能：执行信号量的P操作**
- **输入：整型参数id，需要操作的信号量在信号量数组中索引**
- **输出：成功返回 0；参数非法或检测到死锁返回 -1**

**背景：**  
P 操作用于“获取资源”，当计数不足时应阻塞等待。为避免丢唤醒，休眠使用信号量自身锁。

**设计：**

1. **参数校验**：`id` 在合法范围内。
2. **互斥检查**：持锁检查 `allocated`。
3. **资源获取**：`resource_count--`，不足则准备阻塞。
4. **死锁检测**：阻塞前调用 `deadlock_detect(&sems[id])`，若发现等待环路则回滚计数并返回 `-1`。
5. **阻塞与恢复**：`waiters++` 并在信号量上睡眠，醒来后 `waiters--` 并释放锁。
6. **所有者记录**：仅当 `resource_count == 0` 时记录 `owner`，避免被等待者覆盖。

**关键代码：**

```c
int sys_sem_p()
{
  int id;
  if (argint(0, &id) < 0 || id < 0 || id >= SEM_MAX_NUM) // 参数合法性检查
  {
    return -1;
  }

  acquire(&sems[id].lock); // 获取信号量锁
  if (sems[id].allocated == 0)
  {
    release(&sems[id].lock);
    return -1;
  }

  sems[id].resource_count--;
  if (sems[id].resource_count < 0)
  {
    if (deadlock_detect(&sems[id]))
    {
      sems[id].resource_count++;
      release(&sems[id].lock);
      return -1;
    }
    sems[id].waiters++;
    sleep(&sems[id], &sems[id].lock); // 使用信号量锁进行休眠
    sems[id].waiters--;
  }
  if (sems[id].resource_count == 0)
    sems[id].owner = myproc()->pid;

  release(&sems[id].lock); // 释放信号量锁
  return 0;
}
```

**测试：**

- `semtest`：覆盖阻塞等待与并发计数正确性。

# **define SYS_sem_p_multi 54**

- **功能：对多个信号量执行 AND P 操作，原子获取资源**
- **输入：n（信号量数量），ids（信号量 id 数组）**
- **输出：成功返回 0，失败返回 -1**

**背景：**  
AND 信号量用于解决“多资源同时获取”的场景，避免持有部分资源导致的死锁。典型应用是生产者-消费者中同时需要多个资源（缓冲区 + 互斥锁）。

**设计：**
1. **参数校验与拷贝**：`n` 必须合法；从用户态拷贝 `ids`。
2. **排序与去重**：对 `ids` 排序，检查范围与重复，确保固定加锁顺序避免死锁。
3. **原子判定**：
   - 依次获取所有信号量锁，检查 `allocated` 与 `resource_count`。
   - 若全部可用，统一 `resource_count--` 后释放锁并返回成功。
4. **阻塞等待**：
   - 记录第一个不足的信号量作为 `wait_id`，所有涉及信号量 `waiters++`。
   - 释放除 `wait_id` 外的锁，仅在 `wait_id` 上 `sleep`，唤醒后 `waiters--` 并重试。

**关键代码：**

```c
for (;;) {
  int wait_id = -1;
  for (int i = 0; i < n; i++)
    acquire(&sems[ids[i]].lock);
  for (int i = 0; i < n; i++) {
    if (sems[ids[i]].resource_count <= 0 && wait_id < 0)
      wait_id = ids[i];
  }
  if (wait_id < 0) {
    for (int i = 0; i < n; i++) sems[ids[i]].resource_count--;
    for (int i = 0; i < n; i++) release(&sems[ids[i]].lock);
    return 0;
  }
  for (int i = 0; i < n; i++) sems[ids[i]].waiters++;
  for (int i = 0; i < n; i++) if (ids[i] != wait_id) release(&sems[ids[i]].lock);
  sleep(&sems[wait_id], &sems[wait_id].lock);
  release(&sems[wait_id].lock);
}
```

**测试：**

- `semandtest`：验证 AND 语义下可完成，单信号量顺序易死锁。
- `pcand`：生产者-消费者场景中对比 AND 获取的正确性。

**背景：**  
消息队列是课本“信箱通信”的经典模型。`mqget` 以 key 作为信箱标识，允许多个进程共享同一消息队列。

**设计：**

1. 通过 `key` 查找已有队列；不存在则创建新队列。
2. 使用 `mqmask/refcount` 记录进程对该队列的引用。
3. 全局锁 `mqlock` 保护队列表一致性。

**实现思路：**

1. **获取锁**
  - **调用mqloc锁来保证检查和修改队列的互斥访问**
2. **查找消息队列**
  - **调用findkey(key)，查找是否已存在相应消息队列**
  - **如果没找到则创建一个新队列**
3. **判断进程是否已经使用该消息队列**
  - **如果当前进程未使用该消息队列，则通过 proc->mqmask |= 1 << idx 将该进程标记为使用这个消息队列**
  - **增加引用计数**
4. **消息队列未找到**
  - **调用 newmq(key) 创建一个新的消息队列**
5. **释放锁，并返回**

**关键代码：**

```c

{
    acquire(&mqlock);
    int idx = findkey(key);
    if (idx != -1)
    { // 如果key对应的消息队列已经创建
        if (!(proc->mqmask >> idx & 1))
        {
            proc->mqmask |= 1 << idx; // 标记该进程使用该消息队列
            mqs[idx].refcount++;      // 消息队列的引用计数+1
        }
        release(&mqlock);
        return idx;
    }

    // 对应key消息队列未创建则newmg创建
    idx = newmq(key); // 创建消息队列
    release(&mqlock);
    return idx; // 返回该消息队列在mqs[]中的下标
}
```

# **define SYS_msgsnd 47**

- **功能：消息队列的管理（包括消息存储、进程阻塞与唤醒等）**
- **输入：mqid(消息队列id)、msg(消息数据指针)、sz(消息字节数)**
- **输出：消息发送成功返回0，失败返回-1**

**背景：**  
发送端向信箱投递消息，若队列满则阻塞等待，体现典型生产者行为。

**设计：**

1. **建业消息合法性**
  - **检查消息队列id是否有效**
  - **检查消息队列是否存在**
2. **解析传入的消息数据**
  - **消息数据 msg 的前 4 字节用来存储消息类型，因此通过 msg + 4 获取实际消息内容**
  - **将消息的前 4 字节（即 msg 指向的第一个整数）作为消息类型，保存在 type 中**
3. **检查消息队列是否为空**
4. **获取锁并进行消息发送**
  - **若消息队列有足够空间，则在队尾空闲消息区插入新消息并更新状态**
5. **队列空间不足则阻塞进程**

**关键代码：**

```c
int sys_msgsnd(uint mqid, void *msg, int sz)
{
    // 校验消息队列的合法性
    if (mqid < 0 || mqid >= MQMAX || mqs[mqid].status == 0)
    {
        return -1;
    }

    // 解析传入的消息数据
    char *data = (char *)(((int *)(msg + 4)));

    int *type = ((int *)msg); // 获取消息类型

    // 如果队列为空，打印错误信息
    if (mqs[mqid].msgs == 0)
    {
        printf("msgsnd failed: msgs == 0.\n");
        return -1;
    }

    acquire(&mqlock);

    while (1)
    { // 一直循环直到发送成功
        if (mqs[mqid].curbytes + sz + 16 <= mqs[mqid].maxbytes)
        { // 如果剩余空间充裕
            struct msg *m = mqs[mqid].msgs;

            // 找到队尾最后一个空闲消息区
            while (m->next != 0)
            {
                m = m->next;
            }
            // 退出循环时，m->next == 0，表示空闲消息区

            m->next = (void *)m + m->datasize + 16; // 计算用于存储消息的起始位置
            m = m->next;                            // m为本消息存储空间起点

            m->type = *(type);            // 填写本消息的类型
            m->next = 0;                  // 本消息暂无后续消息
            m->dataaddr = (void *)m + 16; // 数据区的起始位置
            m->datasize = sz;             // 数据长度

            // 复制消息数据到消息区
            memmove(m->dataaddr, data, sz);

            mqs[mqid].curbytes += (sz + 16); // 可用空间缩减

            // 唤醒所有读阻塞进程
            for (int i = 0; i < rstart; i++)
            {
                wakeup(rqueue[i]);
            }

            rstart = 0; // 读阻塞队列置空
            release(&mqlock);
            return 0; // 成功发送消息
        }
        else
        { // 如果空间不足，进程睡眠在wqueue阻塞队列
            printf("msgsnd: cannot alloc: pthread: %d sleep.\n", myproc()->pid);
            wqueue[wstart++] = proc; // 将当前进程加入等待队列

            sleep(proc, &mqlock); // 进入休眠状态，等待唤醒
        }
    }
    return -1; // 发送失败
}
```

- `msgtest`：验证队列满/空阻塞与 type 匹配场景。

# **define SYS_msgrcv 48**

- **功能：从指定的消息队列中接收消息**
- **输入：mqid(消息队列id)、msq(消息缓冲区指针)、sz(目标缓冲区大小)**
- **输出：若成功读取消息返回0，若执行发生错误或消息队列没有匹配消息，返回-1**

**背景：**  
接收端从队列取消息，若队列为空则阻塞，体现“信箱通信”的同步语义。

**设计：**

1. **检验消息队列合法性**
  - **检查消息队列id是否合法**
  - **校检消息队列状态是否有效**
2. **解析消息数据**
  - **消息类型被存储在 msg 的前 4 字节中，因此将 msg 的地址转换为 int* 类型，读取消息类型**
  - **数据部分从 msg + 4 开始，作为消息内容存储**
3. **获取锁并读取消息**
  - **遍历消息队列，查找匹配的消息类型**
  - **找到符合条件的消息，先将内容复制到目标缓冲区**
  - **将消息从队列中移除**
  - **更新当前队列字节数**
  - **调用reloc对消息队列进行内存整理**
  - **唤醒所用因队列满而处于阻塞状态的写进程**
  - **释放锁并返回0**
4. **队列中没用符合的消息时，进入 阻塞**
  - **返回-1**

**关键代码：**

```c
int sys_msgrcv(uint mqid, void *msg, int sz)
{
    // 校验消息队列的合法性
    if (mqid < 0 || mqid >= MQMAX || mqs[mqid].status == 0)
    {
        return -1;
    }

    int *type = msg;     // 待读取消息类型
    int *data = msg + 4; // 待读取消息目标位置

    acquire(&mqlock);

    while (1)
    {
        struct msg *m = mqs[mqid].msgs->next;
        struct msg *pre = mqs[mqid].msgs;

        while (m != 0)
        {
            if (m->type == *type)
            { // 找到要读取的消息类型
                // 复制消息数据到目标位置
                memmove(data, m->dataaddr, sz);

                // 从队列中删除已读取的消息
                pre->next = m->next;

                // 释放消息占用的空间
                mqs[mqid].curbytes -= (m->datasize + 16);

                // 重新整理内存
                reloc(mqid);

                // 唤醒所有写阻塞进程
                for (int i = 0; i < wstart; i++)
                {
                    wakeup(wqueue[i]);
                }
                wstart = 0; // 写阻塞队列置空

                release(&mqlock);
                return 0; // 成功读取消息
            }

            pre = m;
            m = m->next;
        }

        // 如果没有找到匹配的消息类型，当前进程进入休眠状态
        printf("msgrcv: can not read: pthread: %d sleep.\n", proc->pid);

        // 将当前进程加入读阻塞队列
        rqueue[rstart++] = proc;

        sleep(proc, &mqlock); // 进入休眠状态，等待唤醒
        return -1;            // 未能成功读取消息
    }
}
```

```c
    brelse(b);
    return 0; // 成功
}
```

# **define SYS_rt_set 81**

- **功能：设置 LLF 实时调度参数**
- **输入：pid，period，runtime，deadline（单位均为 ticks）**
- **输出：成功返回 0，失败返回 -1**

**背景：**  
为了对齐课本“实时调度”内容，需要对进程设置周期、预算与截止期参数，让调度器按 LLF 选择任务。

**参数语义：**

- `period`：任务周期长度（每个周期会释放一次预算）。
- `runtime`：每周期允许的 CPU 预算（ticks）。
- `deadline`：相对截止期，必须满足 `runtime <= deadline <= period`。

**设计：**

1. **参数校验**：`period/runtime/deadline > 0`，且 `runtime <= deadline <= period`。
2. **状态切换**：将进程标记为 `SCHED_RT_LLF`，写入 `rt_*` 字段。
3. **预算初始化**：设置 `rt_release = now`，`rt_deadline_tick = now + deadline`，`rt_remaining = runtime`。
4. **调度效果**：调度器优先选择 LLF 任务（按松弛度最小策略）。

**错误条件：**

- pid 不存在。
- 参数不合法。

**关键代码：**

```c
uint64 sys_rt_set(void)
{
  int pid, period, runtime, deadline;
  if (argint(0, &pid) < 0 || argint(1, &period) < 0 ||
      argint(2, &runtime) < 0 || argint(3, &deadline) < 0)
    return -1;
  return rt_set(pid, period, runtime, deadline);
}
```

**使用示例：**

```c
// 每 10 ticks 释放预算，需在 6 ticks 内完成 2 ticks 的工作
if (rt_set(getpid(), 10, 2, 6) < 0) {
  printf("rt_set failed\n");
}
```

**测试：**

- `llftest`：对比 normal/rt 模式下 deadline miss 数量。

# **define SYS_mon_create 62 / SYS_mon_free 63 / SYS_mon_enter 64 / SYS_mon_exit 65**

- **功能：** 提供内核级管程（Monitor）的创建与互斥进入/退出接口。
- **输入：** 监视器 id（除 create 外）。
- **输出：** create 返回监视器 id（失败返回 -1）；其它成功返回 0，失败/死锁检测返回 -1。

**背景：**  
管程提供“互斥进入 + 条件同步”的高层抽象，降低并发编程出错概率。

**设计（监视器）：**
1. **全局表管理**：固定大小 `monitors[]`，每个监视器含 `lock/locked/owner/waiters/conds[]`。
2. **创建/释放**：`mon_create` 选择空闲项初始化；`mon_free` 要求未占用且无等待者。
3. **进入/退出**：`mon_enter` 若被占用则先做死锁检测，必要时返回 `-1`；否则在监视器上 `sleep`；`mon_exit` 释放并唤醒一个等待者。
4. **异常退出清理**：`exit()` 中调用 `monitor_release_on_exit`，防止持锁退出导致永久锁死。

**关键代码（进入/退出）：**

```c
while (m->locked) {
  if (myproc()->killed) return -1;
  if (deadlock_detect(m)) return -1;
  m->waiters++;
  sleep(m, &m->lock);
  m->waiters--;
}
m->locked = 1;
m->owner = pid;
// ...
m->locked = 0;
m->owner = 0;
wakeupOneProc(m);
```

# **define SYS_cond_create 66 / SYS_cond_free 67 / SYS_cond_wait 68 / SYS_cond_signal 69 / SYS_cond_broadcast 70**

- **功能：** 提供条件变量（Condition Variable）操作，支持等待、唤醒一个、唤醒全部。
- **语义：** 采用 Mesa 语义：`cond_wait` 会释放监视器并睡眠，被唤醒后重新竞争进入监视器。
- **输入：** 监视器 id 与条件变量 id（create 返回条件变量 id）。
- **输出：** 成功返回 0（create 返回 id），失败返回 -1。

**设计（条件变量）：**
1. **创建/释放**：`cond_create` 在指定监视器内分配条件变量；`cond_free` 要求无等待者。
2. **等待**：`cond_wait` 必须由监视器持有者调用；先释放监视器并睡眠在条件变量上。
3. **唤醒**：`cond_signal` 唤醒一个等待者；`cond_broadcast` 唤醒所有等待者。
4. **重新竞争**：被唤醒后通过 `mon_enter` 逻辑重新获得监视器（Mesa 语义）。

**关键代码（cond_wait）：**

```c
m->conds[cid].waiters++;
m->locked = 0;
m->owner = 0;
wakeupOneProc(m);
sleep(&m->conds[cid], &m->lock);
m->conds[cid].waiters--;
while (m->locked)
  sleep(m, &m->lock);
m->locked = 1;
m->owner = pid;
```
